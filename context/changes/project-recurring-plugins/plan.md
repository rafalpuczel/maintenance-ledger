# Project Recurring Plugins List â€” Implementation Plan

## Overview

Deliver FR-009 (roadmap slice S-05): let a user compose a **per-project recurring plugins list** by picking entries from the global `plugin_catalog` and optionally adding free-text names. A new join table `project_recurring_plugins(project_id, plugin_id)` stores the membership; free-text names are auto-promoted into the catalog (reusing the existing `promoteToCatalog()` hook) and then linked, so every recurring entry is catalog-backed. The composing UI is a React island rendered on the project detail page (`projects/[slug].astro`), mirroring the proven plugins-catalog / pm-contacts CRUD-island pattern. This list is what S-06 will later read to seed a new report's plugin repeater â€” that seeding is **not** in this slice.

## Current State Analysis

The codebase has a mature, repeated vertical-slice pattern (projects, brand-settings, plugins-catalog, pm-contacts), all built identically:

- **Migrations** (`supabase/migrations/`): `<timestamp>_create_<table>.sql`; reuse the shared `public.set_updated_at()` trigger function defined once in `20260529144131_create_projects.sql`; enable RLS with no policies (the Worker uses the `sb_secret_` key which bypasses RLS).
- **Generated types**: `src/types/database.types.ts`, regenerated by `npm run db:types`.
- **Per-domain lib folder** `src/lib/<domain>/`: `schema.ts` (zod), `queries.ts` (PostgREST functions taking a fresh `SupabaseClient<Database>`, mapping Postgres `23505` to a typed error), `form.ts` (`FormData` â†’ validated input, first-issue message).
- **API routes** `src/pages/api/<domain>/`: `index.ts` (POST create), `[id].ts` (POST update), `[id]/delete.ts` (POST delete) â€” all parse form, call a query, then `context.redirect` with `?ok=â€¦` / `?error=â€¦` (no JSON, post-redirect-GET).
- **UI**: an `.astro` page fetches rows server-side and hands them to a React island (`client:load`) that does inline add/edit/delete via native `<form method="POST">` posts.

Key existing assets this slice reuses directly:

- `plugin_catalog` table â€” `id`, `name`, `name_key` (generated `lower(trim(name))`, UNIQUE), `notes`. (`supabase/migrations/20260529190000_create_plugin_catalog.sql`)
- `promoteToCatalog(client, name)` â€” idempotent `upsert({ name }, { onConflict: "name_key", ignoreDuplicates: true })`; skips blank; safe to call blindly. (`src/lib/plugins-catalog/queries.ts:66`) Its migration comment already names S-05 as a consumer.
- `listCatalog(client)` â€” returns all catalog entries ordered by name. (`src/lib/plugins-catalog/queries.ts:20`)
- `projects` table + `getProjectBySlug` (`src/lib/projects/queries.ts:28`).
- The project detail page `src/pages/projects/[slug].astro` currently renders only the edit form + delete button â€” there is **no section below it**, which is exactly where the recurring-list island goes.
- Island building blocks: `FormField` (`src/components/auth/FormField.tsx`), `SubmitButton`, `ServerError`, `Button`, `useFormStatus` row-submit pattern (see `src/components/plugins-catalog/PluginCatalog.tsx`).

### Key Discoveries:

- **The add form must differ from the catalog template in one spot.** Catalog/pm-contacts add a row via a free-text `<input>`. FR-009 is "pick from the catalog (+ optional free-text)", so the recurring island's add control is a **`<select>` of catalog entries** plus a small separate free-text field â€” not a plain text input. Everything else (list fetch, redirect flows, delete-with-confirm) is a direct copy.
- **Two distinct add paths, both end in the same junction insert.** Pick-from-catalog posts a `plugin_id`. Free-text posts a `name`; the route promotes it (`promoteToCatalog`), re-reads its catalog id, then inserts the junction row. Modeling both as one POST endpoint that branches on which field is present keeps the route count identical to the template.
- **Duplicate protection lives in the DB.** A `unique(project_id, plugin_id)` constraint makes re-adding the same plugin a `23505`; the route maps it to a friendly "already on the list" message (a new typed error), matching how `SlugTakenError` / `NameTakenError` are handled elsewhere.
- **Delete needs a composite key, not a surface `id`.** The junction's natural key is `(project_id, plugin_id)`. Give the table its own `id uuid` PK anyway (consistent with every other table + simplest delete route `[id]/delete.ts`), with the unique constraint on the pair.
- **No `updated_at` is needed** â€” a membership row is immutable (you add or remove it, never edit it). Carry only `created_at`. This is the one intentional deviation from the shared-trigger convention, and it means **no `set_updated_at` trigger on this table**.
- **Lessons apply** (`context/foundation/lessons.md`): keep `@typescript-eslint/no-misused-promises` off for `.astro` (already configured); type the island's submit handler as `React.SubmitEvent<HTMLFormElement>` (not deprecated `FormEvent`); judge lint/build by exit code in verification.

## Desired End State

On a project's detail page, below the edit form, a "Recurring plugins" section lists the plugins currently attached to that project (alphabetical). The user can:

- add a plugin by picking it from a dropdown of catalog entries, or
- add a plugin by typing a name not in the catalog (which gets promoted into the catalog, then attached),
- remove any plugin from the list (with an inline confirm),
- and is prevented from adding the same plugin twice (friendly message, no 500).

Verification: create a project, attach two catalog plugins + one free-text name, confirm the free-text name now appears in `/plugins-catalog`, confirm re-adding a plugin shows the "already on the list" message, remove one, reload â€” state persists. `npm run lint`, `npm run build`, and `npm test` pass.

## What We're NOT Doing

- **Seeding a new report** with this list â€” that is S-06 (report-authoring). This slice only delivers list *composition*; it exposes a clean read (`listRecurringPlugins`) for S-06 to consume later.
- Report authoring, repeaters, WP-CLI paste, PDF, email â€” all later slices.
- **Any cadence / due-date / "next due" / overdue / notification concept** â€” explicitly out of scope (this was an early misframing; FR-009 is a membership list, not a scheduler).
- Editing a membership row (rows are add/remove only â€” no `updated_at`, no edit UI).
- Reordering / manual position â€” ordering is alphabetical by plugin name.
- Catalog CRUD changes â€” the catalog already exists; we only *read* it and *promote* into it via the existing hook.
- A separate `/projects/[slug]/recurring` page â€” the UI lives inline on the existing detail page.
- Multi-tenancy / `agency_id` columns (explicit roadmap lock, Open Q3).

## Implementation Approach

Copy the plugins-catalog vertical slice, adapting four things: (1) the table is a join with a composite unique constraint and no `updated_at`; (2) the query layer exposes list/add-by-id/add-by-name/remove plus a typed `AlreadyOnListError`; (3) the single create route branches on `plugin_id` vs `name` (promoting free-text); (4) the island's add form is a catalog `<select>` + a free-text field instead of one text input. Four phases mirror the established commit cadence (p1 table+types, p2 schema+queries+form, p3 routes, p4 UI).

### Changes Required:

#### 1. Migration

**File**: `supabase/migrations/<timestamp>_create_project_recurring_plugins.sql` (timestamp later than `20260529200000`; use a `20260530â€¦` stamp)

**Intent**: Define the projectâ†”plugin join table that stores a project's recurring-plugins membership. Follow every table convention in the repo except `updated_at` (rows are immutable).

**Contract**:

- `alter table â€¦ enable row level security;` with no policies (defense-in-depth, matches every table).

## Phase 2: Validation schema + query layer + form parser

### Overview

Add `src/lib/project-recurring-plugins/` with `schema.ts`, `queries.ts`, `form.ts`, following the plugins-catalog trio.

### Changes Required:

#### 1. Validation schema

**File**: `src/lib/project-recurring-plugins/schema.ts`

**Intent**: Validate the two add shapes â€” a catalog pick (a `plugin_id` UUID) and a free-text add (a non-empty `name`). The free-text name reuses the same "required, trimmed" rule as the catalog's `name`.

**Contract**: Export a `name` schema (`z.string().trim().min(1, "Plugin name is required")`) reused for free-text, and a `plugin_id` schema (`z.string().uuid()`). Export inferred input types. Two small schemas (or one discriminated union) â€” keep it minimal; the route decides which path applies based on which field is present.

#### 2. Query layer

**File**: `src/lib/project-recurring-plugins/queries.ts`

**Intent**: All PostgREST access for the join table plus a typed duplicate error. Each function takes a fresh `Client = SupabaseClient<Database>`.

**Contract**:
- `export type RecurringPlugin` â€” shape returned to the UI: the joined plugin `id`, `name`, `notes`, plus the junction row `id` (needed for the delete route). Define explicitly (not the raw Row) since it's a join projection.
- `class AlreadyOnListError extends Error` â€” message "That plugin is already on this project's list"; constructed when an insert returns `23505`. Mirrors `NameTakenError`.
- `listRecurringPlugins(client, projectId): Promise<RecurringPlugin[]>` â€” select junction rows for the project with the embedded catalog row (PostgREST nested select, e.g. `select("id, plugin:plugin_catalog(id,name,notes)")`), ordered by the plugin name ascending; map to `RecurringPlugin`. **This is the read S-06 will reuse.**
- `addRecurringPluginById(client, projectId, pluginId): Promise<void>` â€” insert `{ project_id, plugin_id }`; map `23505` â†’ `AlreadyOnListError`.
- `addRecurringPluginByName(client, projectId, name): Promise<void>` â€” call `promoteToCatalog(client, name)` (imported from `@/lib/plugins-catalog/queries`), then look up the catalog id by normalized name (`name_key = lower(trim(name))`), then delegate to `addRecurringPluginById`. (Promote is idempotent, so an existing catalog name simply resolves to its id and gets linked.)
- `removeRecurringPlugin(client, id): Promise<void>` â€” delete the junction row by its `id`.

**Contract (snippet â€” the one non-obvious bit, the nested select + ordering by the embedded column):**
```
client
  .from("project_recurring_plugins")
  .select("id, plugin:plugin_catalog(id, name, notes)")
  .eq("project_id", projectId)
  .order("name", { foreignTable: "plugin_catalog", ascending: true });
```

#### 3. Form parser

**File**: `src/lib/project-recurring-plugins/form.ts`

**Intent**: Parse the add form into a discriminated result the route can branch on: a catalog pick vs a free-text add vs invalid.

**Contract**: `parseAddRecurringForm(form: FormData)` returns `{ ok: true; kind: "id"; pluginId } | { ok: true; kind: "name"; name } | { ok: false; message }`. If a `plugin_id` field is present and non-empty, validate it as a UUID (`kind: "id"`); else if a `name` field is present and non-empty, validate it as a name (`kind: "name"`); else return the first validation message (e.g. "Pick a plugin or enter a name"). Follow the first-issue-message convention from `parsePluginCatalogForm`.

## Phase 3: CRUD API routes

### Overview

Add `src/pages/api/project-recurring-plugins/` with a create route (branching on pick vs free-text) and a delete route, both redirecting back to the project detail page.

### Changes Required:

#### 1. Create route

**File**: `src/pages/api/project-recurring-plugins/index.ts`

**Intent**: Accept an add submission, branch on catalog-pick vs free-text, write the junction row, and redirect back to the originating project's detail page with a status param.

**Contract**: `POST` handler. Reads `project_id` and the originating `slug` from the form (the slug is needed to build the redirect target, since redirects go to `/projects/<slug>` â€” mirror how `ProjectForm` passes `returnSlug`). Calls `parseAddRecurringForm`; on `kind:"id"` calls `addRecurringPluginById`, on `kind:"name"` calls `addRecurringPluginByName`. Redirect to `/projects/<slug>?ok=plugin-added` on success; on `AlreadyOnListError` redirect with its message; on other errors a generic "Could not add the plugin". On parse failure redirect with the parse message. All via `context.redirect` (no JSON), matching `api/plugins-catalog/index.ts`.

#### 2. Delete route

**File**: `src/pages/api/project-recurring-plugins/[id]/delete.ts`

**Intent**: Remove one membership row, redirect back to the project detail page.

**Contract**: `POST` handler reading `context.params.id` (junction row id) and the `slug` from the form for the redirect. Calls `removeRecurringPlugin`; redirect `/projects/<slug>?ok=plugin-removed` on success, generic error message otherwise. Mirrors `api/plugins-catalog/[id]/delete.ts` but with the slug-based redirect target.
